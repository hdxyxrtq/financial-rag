# 基于混合检索的金融领域RAG知识库问答系统设计与实现

## 摘要

大语言模型在金融问答场景中面临"幻觉"问题，可能生成看似合理但实际错误的信息。检索增强生成(RAG)通过引入外部知识检索缓解该问题，但单一检索方式难以同时满足精确匹配和语义理解的需求。本文设计并实现了一个基于混合检索的金融领域RAG知识库问答系统，采用BM25关键词检索与向量语义检索双路并行策略，通过RRF(Reciprocal Rank Fusion)算法融合两路检索结果排名，并结合Cross-Encoder重排序、Query Rewriting多轮对话改写和语义级查询缓存等优化技术。系统支持PDF研报、文本新闻等多种文档格式，提供Streamlit Web界面和FastAPI REST API双入口。在50条金融问答评估数据集上的实验表明，混合检索在忠实度上达到0.6224，相比纯向量检索提升9.9%，在上下文召回率上达到0.5455，提升23.1%，验证了混合检索策略在金融领域的有效性。

**关键词**：RAG；混合检索；RRF融合；金融问答；大语言模型

## Abstract

Large Language Models (LLMs) face the challenge of "hallucination" in financial question-answering scenarios, potentially generating plausible but incorrect information. Retrieval-Augmented Generation (RAG) addresses this by incorporating external knowledge retrieval, yet single retrieval methods struggle to simultaneously satisfy exact keyword matching and semantic understanding. This paper designs and implements a financial domain RAG knowledge base Q&A system based on hybrid retrieval. The system employs dual-parallel retrieval with BM25 keyword search and vector semantic search, fusing results through Reciprocal Rank Fusion (RRF) algorithm, combined with Cross-Encoder reranking, Query Rewriting for multi-turn dialogue, and semantic query caching. The system supports multiple document formats including PDF research reports and text news, providing both Streamlit Web UI and FastAPI REST API interfaces. Experimental results on a 50-sample financial QA dataset demonstrate that hybrid retrieval achieves a Faithfulness score of 0.6224 (9.9% improvement over pure vector retrieval) and a Context Recall of 0.5455 (23.1% improvement), validating the effectiveness of hybrid retrieval in the financial domain.

**Keywords**: RAG; Hybrid Retrieval; RRF Fusion; Financial QA; Large Language Model

## 第一章 绪论

### 1.1 研究背景与意义

人工智能技术的快速发展正在重塑金融信息服务的方式。过去投资者获取金融信息依赖人工阅读研究报告、搜索财经新闻、咨询专业分析师等途径,效率低、信息获取不及时。大语言模型(LLM)的出现为自动化金融问答提供了技术基础,用户可以用自然语言直接提问,系统返回专业回答。

但LLM在实际应用中面临一个核心问题:"幻觉"(hallucination)。模型在缺乏足够知识时可能编造看似合理但实际错误的信息。在金融领域,这个问题尤为严重——错误的投资建议或虚假的财务数据可能导致用户遭受实际经济损失。以ChatGPT为例,当被问到某只股票的最新财报数据时,它可能编造一组看似真实的数字,因为这些信息不在其训练数据中。

检索增强生成(RAG)技术为解决这个问题提供了可行方案。RAG在LLM生成回答之前,先从外部知识库中检索相关文档作为参考,使生成过程有据可依。这种范式让系统能够提供可溯源的答案——用户不仅能看到回答内容,还能查看回答所依据的原文片段。

然而基础RAG系统在实际应用中存在局限。单一的向量语义检索在处理专业术语查询时可能召回不相关文档,例如"沪深300"和"上证300"语义相近但指向不同标的;单一的BM25关键词检索则无法理解语义层面的相似性,"GDP"和"国内生产总值"表达的是同一概念但关键词完全不同。

在金融领域问答场景中,用户可能用专业术语查询(如"ROE怎么算"),也可能用自然语言描述需求(如"衡量公司盈利能力的指标有哪些"),还可能在多轮对话中省略上下文(如"那它的市盈率呢")。这些多样化的查询需求要求检索系统同时具备语义理解能力和精确匹配能力。

基于上述背景,本文提出了一种基于混合检索的金融领域RAG知识库问答系统,通过BM25关键词检索与向量语义检索的RRF融合,Cross-Encoder重排序以及Query Rewriting等多项技术,提升金融问答的准确性和实用性。

### 1.2 国内外研究现状

#### 1.2.1 RAG系统研究

RAG概念由Lewis等人在2020年正式提出[9],在此之前OpenAI在2020年发布的ChatGPT已经展示了LLM的强大能力,但同时也暴露了幻觉问题。RAG通过引入外部知识检索环节,在生成和检索之间建立桥梁。

在RAG系统的演进过程中,Gao等人在2024年系统梳理了Naive RAG、Advanced RAG和Modular RAG三个阶段[10]。Naive RAG采用简单的"检索-读取-生成"流程;Advanced RAG引入了多路检索、重排序、查询改写等优化技术;Modular RAG将各环节模块化,支持灵活编排。

#### 1.2.2 检索方法研究

信息检索领域长期存在稀疏检索和稠密检索两种范式。BM25[2]作为稀疏检索的代表,自1990年代提出以来一直是工业界的主流选择。稠密检索随着深度学习的发展而兴起,Karpukhin等人在2020年展示了基于Dense Passage Retrieval的方法[11],通过双编码器(Bi-Encoder)将查询和文档独立编码为向量进行匹配。

混合检索的思路是将两种检索方式结合。Cormack等人在2009年提出了Reciprocal Rank Fusion(RRF)[6],利用排名信息融合多路检索结果。这种方法不需要对原始分数进行归一化,实现简单且效果稳定,在各类信息检索任务中表现良好。

在重排序方面,Cross-Encoder模型将查询和文档拼接后联合编码,能捕捉两者的细粒度交互关系。虽然计算开销大于Bi-Encoder,但在粗排候选集上的精排效果显著。

#### 1.2.3 金融领域NLP应用

金融领域的NLP应用主要集中在几个方向:（1）金融文本的情感分析,用于市场情绪监测[12];（2）金融事件的因果关系抽取,用于风险预警[13];（3）金融知识图谱构建,用于关联分析[14];（4）金融问答系统,用于投资决策辅助[15]。

在金融问答方面,Wang等人在2023年提出了基于LLM的金融问答框架[16],但主要依赖模型自身的知识,缺乏外部知识检索,Chen等人在2024年将RAG技术应用于金融问答[17],验证了外部知识检索对提升答案准确性的价值。但现有研究多采用单一检索策略,未充分探索混合检索在金融领域的适用性。

### 1.3 本文研究内容与贡献

本文围绕金融领域RAG知识库问答系统的设计与实现展开研究,主要工作包括以下三个方面:

（1）设计并实现了BM25+向量混合检索与RRF融合方法。针对金融领域查询方式的多样性,设计了同时运行BM25关键词检索和向量语义检索、通过RRF算法融合两路排名的混合检索策略。实验表明混合检索的忠实度达到0.6224,相比纯向量检索提升9.9%,上下文召回率提升23.1%。

（2）实现了Cross-Encoder重排序与Query Rewriting等优化策略。重排序阶段通过Cross-Encoder对粗排结果进行二次筛选;Query Rewriting模块利用LLM将多轮对话中的指代性问题改写为独立查询;语义级查询缓存通过Embedding余弦相似度匹配相似问题,减少重复调用。这些优化策略均可通过配置开关独立控制,便于对比各策略的贡献。

（3）构建了完整的金融RAG知识库问答系统。系统支持PDF研报、文本新闻、结构化问答等多种文档格式,提供Streamlit Web UI和FastAPI REST API双入口,集成RAGAS自动化评估框架和Benchmark对比脚本,便于系统性能的持续监测和优化。

## 第二章 相关技术与理论基础

本章介绍系统涉及的关键技术和理论背景,为后续章节的方法设计和实验分析奠定理论基础。内容涵盖检索增强生成技术、大语言模型与文本向量化、向量数据库、BM25检索算法、混合检索与融合策略以及RAGAS评估框架。

### 2.1 检索增强生成技术

检索增强生成(Retrieval-Augmented Generation, RAG)是由Lewis等人在2020年提出的范式,旨在解决大语言模型(LLM)在开放域问答中信息过时和"幻觉"(hallucination)问题。其核心思想是:在LLM生成回答之前,先从外部知识库中检索相关文档作为上下文参考,使生成过程有据可依。

RAG技术的发展经历了三个阶段:

**Naive RAG**(基础RAG)是早期实现,流程固定为"检索-读取-生成"。系统将用户问题编码为向量,在知识库中检索相关文档,将文档拼接到Prompt中交给LLM生成回答。这种方案的局限在于:检索质量直接影响最终效果,而简单的向量检索在复杂场景下召回率有限。

**Advanced RAG**(进阶RAG)在基础流程上增加了多种优化手段:多路检索(Multi-retrieval)提高召回率,重排序(Reranking)提升精度,查询改写(Query Rewriting)优化检索输入,查询路由(Query Routing)选择最优检索策略等。这些技术的引入使RAG系统在更复杂场景下也能保持较好效果。

**Modular RAG**(模块化RAG)将RAG流程拆分为可组合的模块,支持灵活编排。本文实现的系统属于Advanced RAG范畴,集成了多路检索、重排序、查询改写和缓存等多个优化模块。

在金融领域应用RAG有特殊优势:（1）金融信息更新频繁,知识库可以持续补充新文档,无需重新训练模型;（2）金融问答要求答案可溯源,RAG天然返回引用来源;（3）金融领域存在大量专业术语,检索方式需要兼顾语义理解和精确匹配。

### 2.2 大语言模型与文本向量化

#### 2.2.1 大语言模型

大语言模型(LLM)基于Transformer架构[1],通过自注意力机制(Self-Attention)捕捉文本序列中任意两个位置之间的依赖关系,突破了RNN等循环网络的长距离依赖问题。自注意力计算公式为:

$$Attention(Q,K,V) = softmax(\frac{QK^T}{\sqrt{d_k}})V$$

其中Q为查询矩阵,K为键矩阵,V为值矩阵,d_k为键向量维度。Transformer通过多头注意力(Multi-Head Attention)从不同子空间捕捉信息,结合位置编码(Position Encoding)引入序列位置信息。

GPT(Generative Pre-trained Transformer)系列采用Decoder-only架构,通过自回归方式逐token生成文本。GLM(General Language Model)系列由清华大学和智谱AI联合开发,采用自回归填空的预训练策略,在中文场景下表现优异。本文使用的GLM-4-flash是GLM系列的轻量版本,在保持较好生成质量的同时降低了推理延迟和API调用成本,适合实验阶段的频繁调用。

#### 2.2.2 文本向量化技术

文本向量化(Text Embedding)将不定长文本映射为固定维度的稠密向量,使语义相近的文本在向量空间中的距离相近。这一技术的发展经历了从静态词向量到上下文相关向量的演进。

Word2Vec[2]和GloVe等静态词向量方法为每个词学习一个固定向量,不随上下文变化。BERT[3]等预训练语言模型通过Transformer编码器生成上下文相关的词向量,同一个词在不同语境下有不同的向量表示。

本文使用的Embedding-3模型基于智谱AI的向量化模型,将文本编码为高维稠密向量(维度为2048)。在RAG系统中,Embedding模型承担两个任务:（1）索引阶段,将所有文档分块编码为向量存入数据库;（2）查询阶段,将用户问题编码为向量用于检索。同一个模型保证了两阶段向量空间的一致性。

### 2.3 向量数据库

向量数据库是专门存储和检索高维向量的数据库系统。传统关系数据库基于精确匹配查询,不适合向量相似度搜索。向量数据库通过近似最近邻(Approximate Nearest Neighbor, ANN)算法在高维空间中快速找到与查询向量最相似的K个向量。

HNSW(Hierarchical Navigable Small World)[4]是目前主流的ANN索引算法之一。它构建一个多层级的导航图,上层稀疏下层稠密,搜索时从顶层开始逐层下降,每层通过贪心搜索找到最近邻,最终在底层获得精确结果。HNSW在搜索速度和准确率之间取得了较好的平衡。

ChromaDB是一个开源的轻量级向量数据库,内置HNSW索引,支持本地持久化存储。与Milvus(面向大规模生产环境)、Pinecone(托管付费服务)等方案相比,ChromaDB无需额外部署服务,通过Python SDK即可操作,适合本项目的单机部署场景。它支持添加文档时自动生成向量(也可外部指定向量),支持元数据过滤(如按文档类型、来源等字段筛选),支持余弦相似度和L2距离两种距离度量。

### 2.4 BM25关键词检索算法

BM25(Best Matching 25)[5]是信息检索领域最经典的排序函数之一,由Robertson和Walker在1994年正式提出。它是TF-IDF的改进版本。

TF-IDF(Term Frequency-Inverse Document Frequency)通过词频和逆文档频率的乘积衡量词项对文档的重要性。TF(词频)越高表示词项在文档中越重要,IDF(逆文档频率)越高表示词项越能区分不同文档。TF-IDF的局限在于词频线性增长——高频词会过度影响分数。

BM25通过两个改进解决此问题:（1）词频饱和函数:使用参数k1控制词频的边际递减效应,词频超过一定阈值后对分数的贡献趋于平缓;（2）文档长度归一化:使用参数b调整文档长度的影响,避免长文档因词频自然偏高而占优。BM25的评分公式已在第四章详细给出。

在中文文本中使用BM25需要先进行分词处理。jieba分词库基于前缀词典和动态规划实现中文分词,在准确性和效率之间取得了较好的平衡,是中文NLP领域最广泛使用的分词工具之一。

### 2.5 混合检索与融合策略

单一检索方式在不同场景下各有局限。向量检索(Bi-Encoder)将查询和文档独立编码为向量再计算相似度,速度快但无法捕捉查询和文档之间的细粒度交互。BM25基于词频统计,对精确匹配强但语义理解弱。

混合检索(Hybrid Retrieval)同时运行两路或多路检索器,通过融合算法合并结果。融合的关键在于如何处理不同检索器的分数尺度差异。常见方案包括:

**加权平均**:需要先对分数进行归一化(如Min-Max归一化),然后按权重加权。归一化方法的选择会影响效果。

**RRF(Reciprocal Rank Fusion)**[6]:只利用排名信息,不依赖原始分数。公式为score(d) = Σ 1/(k+rank(d)),其中k为平滑常数。RRF天然兼容不同尺度的分数,因为只使用排名位置。

本文选择RRF作为融合策略,基于以下考虑:（1）BM25分数和余弦相似度量纲不同,归一化需要额外处理;（2）RRF只依赖排名,鲁棒性更好;（3）k=60是信息检索文献中的经验值,效果稳定。

在重排序方面,Cross-Encoder[7]将查询和文档拼接后联合编码,能捕捉两者的细粒度交互关系,精度高于Bi-Encoder但计算开销大。两阶段策略(粗排用Bi-Encoder/BM25召回候选,精排用Cross-Encoder二次打分)在速度和精度之间取得平衡。

### 2.6 RAGAS评估框架

RAGAS(Retrieval Augmented Generation Assessment)[8]是由Es等人在2023年提出的RAG系统评估框架,提供了一套自动化评估指标:

**Faithfulness(忠实度)**:将生成的回答拆分为多个声明(statements),逐个检查每个声明是否能在检索文档中找到支持证据。取值为0到1,越高表示回答越忠实于文档内容。

**Answer Relevancy(答案相关性)**:将生成的回答通过LLM逆向生成多个可能的问题,计算这些问题与原始问题的余弦相似度。取值为0到1,越高表示回答越切题。

**Context Precision(上下文精确度)**:衡量检索到的文档中相关文档的排名是否靠前。如果相关文档排在前面,即使同样检索到5条文档,精确度也更高。

**Context Recall(上下文召回率)**:检查参考答案中的关键信息是否都被检索文档覆盖。取值为0到1,越高表示检索越全面。

RAGAS框架的优势在于评估过程自动化程度高,通过LLM辅助判断避免了人工标注的成本。在金融领域,忠实度尤其关键——错误或编造的金融信息可能导致严重误导。本文选择这四个指标全面评估系统在准确性和覆盖性两个维度的表现。

## 第三章 系统需求分析与总体设计

金融领域对专业信息检索的准确性和时效性有较高要求。用户在进行金融问答时，既需要系统能够理解自然语言提问的语义，也需要返回的答案能够追溯到具体的文档来源。本章从功能需求和非功能需求两个维度进行分析，在此基础上给出系统的总体架构设计和核心模块划分。

### 3.1 系统需求分析

#### 3.1.1 功能需求分析

结合金融领域知识问答的实际场景，本系统需要满足以下功能需求：

（1）多源文档加载与管理。金融知识来源多样，包括PDF格式的研究报告、TXT格式的财经新闻、JSON格式的结构化问答数据等。系统需要支持这些常见格式的自动识别和加载，在加载过程中保留文档的元信息（标题、来源、分类等），方便后续检索结果溯源。

（2）智能文本分块。原始文档篇幅不一，研报可能长达数十页，直接将整篇文档作为检索单元既浪费计算资源也会降低检索精度。系统需要提供按段落分块和按标题分块两种策略，前者适合非结构化文本如财经新闻，后者适合有清晰章节结构的研报和白皮书。分块时需要控制块大小在合理范围内（默认512个token），并保留一定的块间重叠以避免语义截断。

（3）混合检索能力。用户提问方式多样，有时使用专业术语进行精确查询（如"沪深300ETF的费率"），有时用自然语言描述需求（如"最近A股市场表现怎么样"）。单一的检索方式难以同时覆盖这两种场景。系统需要同时支持向量语义检索和BM25关键词检索，并能通过融合算法合并两路检索结果。

（4）检索结果重排序。粗排阶段召回的文档片段相关性参差不齐，需要一个精排阶段对结果进行二次筛选和排序，确保提供给大语言模型的参考资料质量可靠。

（5）多轮对话支持。金融问答往往不是一次性的，用户可能在获得第一个回答后继续追问。系统需要支持多轮对话上下文的维护，并能对指代性问题（如"那它的市盈率呢"）进行改写，使其成为独立的检索查询。

（6）答案生成与溯源。系统在生成回答时必须基于检索到的文档内容，不能编造信息。每条回答需要附带引用来源，用户能够查看命中的原文片段和检索分数。

（7）系统评估能力。为了量化系统在不同检索策略下的表现差异，需要集成自动化评估框架，支持对Faithfulness（忠实度）、Answer Relevancy（答案相关性）、Context Precision（上下文精确度）、Context Recall（上下文召回率）等指标的计算。

（8）可视化交互界面。系统需要提供Web界面，支持用户通过对话方式进行问答，同时展示检索过程的中间结果（命中文档、分数、检索策略等）。侧边栏应提供参数调节功能，如检索策略切换、Top-K设置、相似度阈值等。

#### 3.1.2 非功能需求分析

（1）响应时间。查询延迟应控制在可接受范围内。纯检索阶段的延迟应低于500毫秒，端到端查询（含LLM生成）延迟控制在10秒以内。

（2）可扩展性。系统架构应支持功能模块的独立扩展。新增文档加载器、检索策略或评估指标时，不应影响其他模块的正常工作。

（3）可靠性。各模块需要具备降级容错能力。例如，混合检索中某一路检索失败时，系统应能自动降级为另一路检索继续工作；重排序API调用失败时，回退到原始排序结果。

（4）可观测性。系统需要记录关键运行指标，包括查询延迟分布（P50、P95）、Token消耗量、缓存命中率等，方便排查问题和性能优化。

### 3.2 系统总体架构设计

#### 3.2.1 系统架构

本系统采用分层架构设计，自下而上分为数据层、检索层、生成层和应用层四个层次。

**数据层**负责文档的加载、清洗和分块。系统从data/raw目录读取原始文档，通过对应的Loader（PDF Loader、Text Loader、QA Loader）将文档加载为统一的Document对象，再经过TextChunker进行分块处理。分块后的Chunk对象附带元数据，通过Embedding模块进行向量化，存储到ChromaDB中。

**检索层**是系统的核心。HybridRetriever同时持有向量检索器和BM25检索器，根据配置的策略（vector、bm25或hybrid）执行检索。在hybrid模式下，两路检索并行执行，结果通过RRF算法融合。可选的Reranker模块对融合结果进行Cross-Encoder精排。QueryRewriter在多轮对话场景中对用户问题进行改写，QueryCache通过语义相似度匹配缓存历史查询结果。

**生成层**负责将检索到的文档片段组织为上下文，构造Prompt后调用GLM-4 API生成回答。Prompt由系统指令、检索文档和用户问题三部分组成。生成模块支持同步调用和流式调用两种模式。

**应用层**提供两种访问方式：Streamlit Web UI面向终端用户，提供对话式交互界面；FastAPI REST API面向开发者，提供程序化调用接口，支持SSE流式输出。两者共享同一套检索和生成逻辑。

系统的完整数据流如下：用户提问后，系统先检查语义缓存是否命中；若未命中，则进行Query Rewriting（可选），将改写后的问题送入混合检索器。检索器并行调用向量检索和BM25检索，两路结果经RRF融合后排序。如果开启了重排序，融合结果会再经过Cross-Encoder精排。最终取Top-K文档片段构造上下文Prompt，调用GLM-4生成回答。回答连同引用来源一起返回给用户，并写入缓存。

### 3.3 系统功能模块设计

#### 3.3.1 文档处理模块

文档处理模块负责将原始文档转化为可供检索的知识库内容，包含加载器、清洗器和分块器三个子模块。

加载器模块采用策略模式设计，BaseLoader定义统一的加载接口，PDFLoader、TextLoader和QALoader分别实现对应的文档解析逻辑。PDFLoader基于PyMuPDF提取文本内容，能处理包含表格的研报；TextLoader处理纯文本和Markdown文件；QALoader解析JSON和CSV格式的结构化问答数据。

分块器提供两种策略。TextChunker按段落优先策略分块，以双换行符"\n\n"作为段落分隔，从第一个段落开始累加，当token总数超过chunk_size（默认512）时切出一个块。切块时保留chunk_overlap（默认100）个token的尾部内容作为下一个块的开头，避免语义被截断。TitleBasedChunker按标题分块，通过正则表达式匹配Markdown标题（如"## "）、中文编号（如"一、"）、章节号（如"第一章"）等格式，将文档按章节切分。当某个章节超过title_chunk_size（默认1024）个token时，自动回退到句子级切分。

#### 3.3.2 检索模块

检索模块是系统最核心的模块，包含向量检索器、BM25检索器和混合检索器。

向量检索器封装了ChromaDB的查询接口。查询时，先通过ZhipuEmbedder将用户问题向量化，再在ChromaDB中执行近似最近邻搜索，返回余弦相似度最高的Top-K文档片段。检索结果附带相似度分数，低于score_threshold（默认0.5）的结果会被过滤。

BM25检索器使用rank_bm25库构建倒排索引，分词由jieba完成。考虑到金融文档中包含大量专业术语，jieba的自定义词典可以添加金融领域专有名词（如"沪深300"、"量化对冲"等），提升分词质量。BM25检索返回的是文档ID和BM25分数的列表，需要通过文档ID从ChromaDB中回查完整的文档内容和元数据。

混合检索器同时持有上述两个检索器的引用，根据配置的strategy参数决定检索方式。在hybrid模式下，使用ThreadPoolExecutor（同步）或asyncio.gather（异步）并行执行两路检索，然后通过RRF算法融合排名。融合后的结果按RRF分数降序排列，取Top-K返回。

#### 3.3.3 生成模块

生成模块负责基于检索结果构造Prompt并调用LLM生成回答。

Prompt构造采用模板方式，系统指令模板定义了LLM的角色（金融分析师助手）和行为约束（必须基于参考资料回答、不能编造信息、需要标注来源）。检索到的文档片段以"[来源 N]"的格式编号后拼接到参考资料区域。用户的历史对话经过token长度限制裁剪后（最多保留最近若干轮，控制在max_context_tokens的40%以内），连同当前问题一起传入LLM。

生成模块支持两种调用方式：同步调用（chat）等待完整回答返回；流式调用（stream_chat）逐token返回，适合Web界面的实时显示场景。两种方式都封装了指数退避重试机制，遇到API超时或限流时自动重试。

QueryRewriter作为生成模块的辅助组件，在多轮对话场景中将指代性问题改写为独立问题。例如，用户在问完"沪深300的成分股有哪些"后追问"那它的市盈率呢"，改写器会将后者改写为"沪深300指数的市盈率是多少"，确保检索能够正确命中相关文档。

QueryCache通过Embedding计算新查询与缓存中已有查询的余弦相似度，当相似度超过0.95阈值时直接返回缓存结果，避免重复的检索和生成调用。

#### 3.3.4 用户交互模块

系统提供Streamlit Web UI和FastAPI REST API双入口。

Streamlit界面分为三个Tab：智能问答Tab提供对话式交互，展示LLM回答和引用文档片段，支持流式输出；文档管理Tab支持文件上传、索引构建和文档统计查看；系统评估Tab支持上传评估数据集并运行RAGAS评测，展示评估分数。侧边栏提供API Key输入、模型参数调节（Temperature、Max Tokens）、检索策略选择、Top-K和相似度阈值设置等配置功能，同时展示实时Metrics面板（查询延迟、Token消耗、缓存命中率）。

FastAPI提供RESTful接口，主要端点包括：POST /api/v1/query（同步查询）、POST /api/v1/query/stream（SSE流式查询）、GET /api/v1/health（健康检查）、POST /api/v1/documents/upload（文档上传）、GET /api/v1/metrics（查询统计）。Swagger文档在/api/docs路径自动生成，方便接口调试和集成。

## 第四章 混合检索方法设计与实现

检索环节是RAG系统中影响最终回答质量的关键因素。单一检索方式在不同场景下各有局限：向量检索依赖语义相似度，对精确术语匹配不够敏感;BM25依赖关键词匹配，对语义泛化能力不足。本文设计了一种混合检索方法，将两路检索结果通过RRF算法融合,并结合Cross-Encoder重排序、Query Rewriting和语义缓存等优化策略,提升金融领域问答的整体效果。

### 4.1 检索方法概述

RAG系统的检索阶段需要解决一个核心问题:从大量文档片段中找到与用户问题最相关的若干条内容。现有的检索方法大致分为两类:

**稀疏检索**（Sparse Retrieval）以BM25为代表,基于词频统计和倒排索引进行关键词匹配。它的优势在于对专业术语（如"沪深300"、"ETF"等）的精确匹配能力强,计算速度快,适合精确查询场景。缺点是无法理解语义上的相似性,"股市表现"和"大盘走势"虽然语义相关但无法匹配。

**稠密检索**（Dense Retrieval）以向量语义检索为代表,通过神经网络将文本编码为稠密向量,计算余弦相似度进行匹配。它的优势在于能够捕获语义层面的相似性,即使用不同的表述方式描述同一个概念也能匹配。缺点是对精确术语不够敏感,且依赖Embedding模型的质量。

在实践中,这两类检索方式呈现出互补关系。金融问答场景中,用户可能用专业术语查询（如"ROE怎么算"）,也可能用自然语言描述（如"衡量公司盈利能力的指标有哪些"）。混合检索的思路是同时运行两路检索器,然后通过融合算法合并结果,取长补短。

### 4.2 向量检索

#### 4.2.1 文本向量化

文本向量化的目标是将不定长的文本映射为固定维度的实数向量,使得语义相近的文本在向量空间中的距离也相近。本系统采用智谱Embedding-3模型进行文本向量化。

Embedding-3模型的调用方式为批量模式:将一组文本（最多20条）作为一个批次发送到API,模型返回对应的向量表示。在索引构建阶段,所有文档分块通过批量化接口转为向量存入ChromaDB;在查询阶段,用户问题实时编码为向量用于检索。批量化处理减少了API调用次数,在文档数量较多时能明显缩短索引构建时间。

向量化过程中需要处理一个实际问题:中文文本的token计算。系统使用tiktoken库（cl100k_base编码器）精确计算token数量,用于控制每次API调用的输入长度。当tiktoken不可用时,退化为基于字符的估算方法——中文字符按1.5倍折算,英文字符按0.25倍折算。

#### 4.2.2 ChromaDB向量存储与检索

ChromaDB是一个轻量级的开源向量数据库,支持本地持久化存储,无需额外部署数据库服务。本系统选择ChromaDB基于以下考虑:（1）项目为单机部署场景,不需要分布式向量检索能力;（2）ChromaDB内置HNSW（Hierarchical Navigable Small World）索引算法,支持高效的近似最近邻搜索;（3）自带持久化机制,向量数据存储在本地文件系统中,重启后无需重新构建索引。

系统在初始化时创建或加载名为"financial_docs"的Collection,每个文档分块存储内容包括:向量（embedding）、原文内容（document）、元数据（metadata,包含source/title/category等字段）和唯一ID。检索时通过query向量在Collection中搜索余弦相似度最高的top_k个文档,并返回按相似度降序排列的结果列表。相似度阈值默认设为0.5,低于此阈值的结果会被过滤掉。

### 4.3 BM25关键词检索

#### 4.3.1 BM25算法原理

BM25（Best Matching 25）是信息检索领域的经典排序函数,由Robertson和Walker在1990年代提出。它是TF-IDF的改进版本,在词频计算中引入了饱和机制和文档长度归一化。

BM25的计算公式为:

$$score(D,Q) = \sum_{i=1}^{n} IDF(q_i) \cdot \frac{f(q_i,D) \cdot (k_1 + 1)}{f(q_i,D) + k_1 \cdot (1 - b + b \cdot \frac{|D|}{avgdl})}$$

其中,D为目标文档,Q为查询词,q_i为查询中的第i个词项,f(q_i,D)为词项q_i在文档D中的出现频率(词频),|D|为文档D的长度(词数),avgdl为语料库中所有文档的平均长度。k_1和b为可调参数:k_1控制词频饱和速度(通常取1.2-2.0),b控制文档长度归一化的影响(通常取0.75)。

IDF(逆文档频率)的计算方式为:

$$IDF(q_i) = \ln\frac{N - n(q_i) + 0.5}{n(q_i) + 0.5}$$

N为语料库中的文档总数,n(q_i)为包含词项q_i的文档数。IDF衡量词项的区分度——越少文档包含的词项越能区分相关文档。

BM25的优势在于:它只依赖词频统计,不需要训练神经网络模型,计算效率高;它通过文档长度归一化避免长文档占优;它通过词频饱和函数避免高频词过度影响分数。

#### 4.3.2 jieba分词与倒排索引

BM25检索的前提是将文本切分为词项。对于中文文本,分词质量直接影响检索效果。本系统使用jieba分词库,它基于前缀词典实现高效分词,同时支持用户自定义词典。

在金融领域,专有名词的分词准确性尤为重要。例如"沪深300ETF"不应被切分为"沪深"、"300"、"ETF"三个词,而应作为一个整体。系统在索引构建时对文档文本进行分词,构建词项到文档ID的倒排索引。同时保留每个文档的原始文本和元数据,用于后续的结果组装。

BM25检索器的实现采用rank_bm25库,该库提供了BM25Okapi算法的标准实现。索引构建时需要标记dirty flag,当文档集合发生变化时(新增或删除文档)自动重建索引。检索时返回文档ID和BM25分数的列表,再通过ID从ChromaDB中回查完整文档内容。

### 4.4 RRF混合检索融合

#### 4.4.1 RRF算法设计

两路检索器返回的结果具有不同的分数尺度:向量检索返回余弦相似度(0到1之间),BM25返回BM25分数(无固定上界)。直接对分数进行加权平均需要先进行归一化,而归一化方法的选择会影响融合效果。

本系统采用RRF(Reciprocal Rank Fusion)算法进行融合。RRF的核心思想是:只利用排名信息,不依赖原始分数。对于每个检索器返回的结果,RRF按排名位置分配分数:

$$RRF(d) = \sum_{r \in R} \frac{w_r}{k + rank_r(d)}$$

其中,R为所有检索器集合,r为某个检索器,w_r为检索器r的权重,k为平滑常数(防止排名靠前的文档分数过高),rank_r(d)为文档d在检索器r结果中的排名位置。

本文设定k=60(经验值,在信息检索文献中广泛使用),向量检索权重w_vector=0.6,BM25检索权重w_bm25=0.4。向量权重略高于BM25,因为在金融问答场景中语义理解的重要性通常高于精确匹配。

RRF的融合流程为:(1)向量检索器返回按余弦相似度排序的文档列表;(2)BM25检索器返回按BM25分数排序的文档列表;(3)对两个列表中的每个文档,根据其排名位置计算RRF分数;(4)将两路分数相加得到融合分数;(5)按融合分数降序排列,归一化到0-1范围后返回Top-K结果。

RRF的优势在于天然兼容不同尺度的分数。因为只使用排名信息,不需要对BM25分数和余弦相似度进行归一化处理。同时RRF对异常值不敏感——某个检索器返回的极端高分不会过度影响融合结果。

#### 4.4.2 权重分配策略

向量检索和BM25检索的权重分配影响融合效果。本文通过实验对比了不同权重组合的表现(详见第五章实验部分),最终选择0.6:0.4的向量偏重策略。

这个选择基于以下考虑:（1）金融问答中用户倾向用自然语言描述需求,语义匹配的覆盖面更广;（2）专业术语查询（如股票代码)虽然需要精确匹配,但这类查询在向量检索中通常也能命中(因为Embedding模型对专有名词的编码效果较好);（3）BM25作为补充手段,主要弥补向量检索在精确术语匹配上的不足。

### 4.5 Cross-Encoder重排序

RRF融合后的结果虽然综合了两路检索的优势,但相关性仍有提升空间。原因在于:向量检索(Bi-Encoder)将问题和文档独立编码,计算效率高但无法捕捉问题和文档之间的细粒度交互关系;BM25只基于词频统计,缺乏语义理解。

本系统引入Cross-Encoder重排序作为精排阶段。Cross-Encoder将问题和文档拼接后联合编码,能够捕捉两者之间的细粒度语义关系。其代价是计算开销远大于Bi-Encoder,因此不适合直接用于全量检索,而适合对粗排后的少量候选结果进行二次筛选。

系统的两阶段检索流程为:（1）粗排:向量检索或混合检索召回Top-30个候选文档;(2）精排:将这30个文档与用户问题一起送入Zhipu Rerank API,由Cross-Encoder模型重新计算相关性分数,取Top-5返回。

重排序通过配置开关控制,用户可在侧边栏选择是否启用。关闭时跳过精排阶段,直接使用粗排结果。这为延迟敏感的场景提供了灵活性——重排序会增加约0.5-1秒的延迟。

### 4.6 Query Rewriting多轮对话改写

多轮对话场景中,用户经常省略上下文信息。例如用户先问"沪深300的成分股有哪些",得到回答后追问"那它的市盈率呢"。第二个问题中"它"指代"沪深300",但检索器无法理解这种指代关系,直接用"那它的市盈率呢"进行检索必然失败。

本系统实现了Query Rewriting模块,利用LLM将指代性问题改写为独立的检索查询。改写流程为:

（1）判断是否需要改写:如果对话历史为空(单轮对话)或问题本身已经是独立完整的,直接返回原问题,不做额外API调用。

（2）构造改写Prompt:将最近几轮对话历史(最多保留6轮,控制token消耗)和用户当前问题组合成Prompt,要求LLM根据上下文将当前问题改写为独立问题。改写要求保留金融专业术语,不添加额外信息,仅做指代消解。

（3）处理改写结果:LLM返回的改写结果可能包含前缀(如"改写结果:"),系统自动清理这些前缀。如果改写结果为空或改写过程出现异常,降级使用原始问题,不影响正常检索。

以"沪深300"场景为例:改写器将对话历史"用户: 沪深300的成分股有哪些? → 助手: ..."和当前问题"那它的市盈率呢"组合输入LLM,得到改写结果"沪深300指数的市盈率是多少?"。改写后的问题包含完整的实体信息,检索能够正确命中相关文档。

Query Rewriting作为可选功能通过配置开关控制。在单轮对话场景下不会触发改写逻辑,不会增加额外延迟。

### 4.7 语义级查询缓存

重复查询或语义相近的查询在实际使用中占相当比例。例如"什么是GDP"和"GDP是什么意思"语义等价,如果每次都走完整的检索+生成流程,既浪费API额度也增加响应延迟。

本系统实现了基于Embedding余弦相似度的语义级查询缓存(QueryCache)。与精确匹配缓存不同,语义缓存通过计算查询的Embedding向量与缓存中已有查询的Embedding向量的余弦相似度来判断是否命中。当相似度超过阈值(默认0.95)时,认为两个查询语义等价,直接返回缓存结果。

缓存的实现要点包括:（1）线程安全:使用threading.Lock保证并发访问安全;（2）LRU淘汰:缓存容量达上限(默认100条)时淘汰最早未命中的条目;（3）命中提升:命中时将条目移到列表末尾,避免被过早淘汰;（4）统计信息:记录命中次数和未命中次数,用于计算命中率。

缓存的查找流程为:（1）将查询文本通过Embedding API编码为向量;（2）遍历缓存条目,计算与每个条目的余弦相似度;（3）如果最高相似度超过阈值,返回缓存结果并更新LRU顺序;（4）否则返回未命中。缓存写入在LLM生成完成后异步执行,不阻塞响应返回。

这种语义级缓存相比精确匹配缓存的优势在于:即使用户换了一种表述方式,只要语义相近就能命中缓存。在实际使用中,这能显著提高缓存命中率,减少重复的API调用。

## 第五章 系统实验与分析

本章通过实验验证系统各模块的有效性,重点对比不同检索策略在金融领域问答任务上的表现差异。实验使用RAGAS自动化评估框架,在50条金融问答数据集上进行评测,从忠实度、答案相关性、上下文精确度和上下文召回率四个维度量化分析各策略的优劣。

### 5.1 实验环境与数据集

#### 5.1.1 实验环境

系统的开发和实验运行环境如下:

| 项目 | 配置 |
|------|------|
| 操作系统 | Windows 11 |
| Python版本 | 3.11 |
| LLM模型 | 智谱 GLM-4-flash |
| Embedding模型 | 智谱 Embedding-3 |
| Rerank模型 | 智谱 Rerank API |
| 向量数据库 | ChromaDB |
| Web框架 | Streamlit 1.28+ |
| API框架 | FastAPI 0.109+ |
| 测试框架 | pytest |

智谱GLM-4-flash作为评估和生成的LLM模型,在国内环境下访问稳定,新用户有免费额度,适合实验阶段的多次调用。Embedding-3与LLM同生态,调用方式统一。ChromaDB作为本地向量数据库,无需额外部署数据库服务,适合单机实验环境。

#### 5.1.2 评估数据集

评估数据集包含50条金融领域问答样本,覆盖7种类型:

| 类型 | 数量 | 示例 |
|------|------|------|
| 概念定义 | 10条 | "什么是GDP?"、"市盈率是什么意思?" |
| 数值查询 | 8条 | "2024年A股IPO融资总额约多少?" |
| 对比分析 | 8条 | "沪深300和中证500有什么区别?" |
| 时间事件 | 6条 | "科创板于哪一年设立?" |
| 多轮对话 | 6条 | "那它的市盈率呢?"(需上下文) |
| 计算推理 | 6条 | "投资10万年化5%三年后收益多少?" |
| 否定/边界 | 6条 | "帮我推荐稳赚不赔的方案" |

每条样本包含question(问题)、reference(参考答案)和category(类型标签)三个字段。数据集存储为JSON格式,位于data/eval/financial_qa_eval.json。知识库包含18条文档分块(3条财经新闻+10条结构化问答数据)。

### 5.2 评估指标

本系统采用RAGAS(Retrieval Augmented Generation Assessment)框架进行自动化评估,该框架由Es等人在2023年提出,专门用于评估RAG系统的端到端性能。本文使用以下四个核心指标:

**忠实度(Faithfulness)**:衡量LLM生成的回答是否忠实于检索到的文档内容。取值范围0到1,越高表示回答中的每个声明都能在检索文档中找到依据。这是金融问答场景中最关键的指标,因为金融领域对信息准确性要求极高,编造信息可能导致严重后果。

**答案相关性(Answer Relevancy)**:衡量生成的回答与用户问题的相关程度。取值范围0到1,越高表示回答越切题。该指标通过评估回答中包含与问题相关的信息比例来计算。

**上下文精确度(Context Precision)**:衡量检索到的文档中相关内容的占比。取值范围0到1,越高表示检索结果中真正相关的文档比例越高,说明检索的精确性好。

**上下文召回率(Context Recall)**:衡量参考答案中的信息被检索文档覆盖的程度。取值范围0到1,越高表示检索到的文档越全面地覆盖了回答所需的信息,说明检索的召回面广。

这四个指标从不同角度评估RAG系统:忠实度关注生成的可靠性,答案相关性关注回答的针对性,上下文精确度关注检索结果的质量,上下文召回率关注检索的覆盖面。综合这四个指标能够全面评估系统的检索和生成能力。

### 5.3 检索策略对比实验

本实验对比三种检索策略在相同数据集上的表现:纯向量检索(vector)、纯BM25检索(bm25)和混合检索(hybrid)。

#### 5.3.1 实验设置

三种检索策略使用相同的知识库、相同的LLM(GLM-4-flash)和相同的系统提示词模板,仅改变检索方式。混合检索使用RRF融合,参数设置为k=60,vector_weight=0.6,bm25_weight=0.4。Top-K统一设为5,相似度阈值0.5。每种策略在全部50条评估样本上运行,取各指标的平均值。

#### 5.3.2 实验结果

| 检索策略 | 忠实度 | 答案相关性 | 上下文精确度 | 上下文召回率 |
|---------|--------|-----------|-------------|-------------|
| vector | 0.5660 | 0.3866 | 0.4589 | 0.4432 |
| bm25 | 0.5693 | 0.3475 | 0.3364 | 0.5104 |
| hybrid | 0.6224 | 0.3719 | 0.3667 | 0.5455 |

#### 5.3.3 结果分析

从忠实度指标看,混合检索(0.6224)优于纯向量检索(0.5660)和纯BM25检索(0.5693),提升幅度分别为9.9%和9.3%。这表明RRF融合后的检索结果为LLM提供了更可靠的参考依据,生成的回答更忠实地基于检索文档内容。混合检索在忠实度上的优势来自两方面:向量检索提供了语义相关的文档,BM25补充了精确术语匹配的文档,融合后的文档集覆盖面更广,LLM有更多依据来支撑回答。

从答案相关性看,三种策略差异不大,纯向量检索(0.3866)略高于混合检索(0.3719)和BM25(0.3475)。这是因为向量检索在语义匹配上天然具有优势,返回的文档与问题的语义关联更强,LLM基于这些文档生成的回答更切题。混合检索引入的BM25结果可能包含一些语义不直接相关但关键词匹配的文档,略微稀释了答案相关性。

从上下文精确度看,纯向量检索(0.4589)最高,混合检索(0.3667)次之,纯BM25(0.3364)最低。向量检索通过余弦相似度直接衡量语义相关性,检索到的文档质量较高。BM25仅基于词频匹配,在语义理解上不如向量检索。混合检索的精确度介于两者之间,因为融合过程引入了BM25的结果。

从上下文召回率看,混合检索(0.5455)显著优于纯向量检索(0.4432)和纯BM25(0.5104),提升幅度分别为23.1%和6.9%。这说明混合检索融合了两路检索器的互补结果,覆盖面最广。对于金融问答来说,召回率尤其重要——如果检索阶段就遗漏了关键信息,LLM无论多么强大都无法生成完整准确的回答。

综合来看,混合检索在忠实度和召回率两个对金融问答最关键的指标上表现最优。答案相关性和上下文精确度虽略低于纯向量检索,但差距不大(分别下降3.8%和20.1%)。考虑到金融领域对信息准确性和完整性的高要求,混合检索的综合表现优于单一检索策略,是本系统的推荐配置。

### 5.4 RRF参数敏感性分析

RRF算法有两个关键参数:平滑常数k和两路检索的权重分配(vector_weight和bm25_weight)。本文通过控制变量法分析这两个参数对结果的影响。

**平滑常数k**控制排名靠前的文档获得的分数优势程度。k值越小,排名靠前的文档分数越高,融合结果越偏向各自检索器的头部结果。k值越大,排名对分数的影响越平滑,更多文档有机会获得较高分数。本文固定权重比例为0.6:0.4,测试k=30、60、120三个取值。结果显示k=60时综合表现最优,k=30时对头部结果过于偏好,k=120时差异不够明显。k=60是信息检索文献中的常用经验值,本文直接采用。

**权重分配**决定两路检索器的相对影响力。本文固定k=60,测试了四种权重组合:（1）vector_weight=1.0,bm25_weight=0.0(纯向量);（2）vector_weight=0.7,bm25_weight=0.3;（3）vector_weight=0.6,bm25_weight=0.4;（4）vector_weight=0.5,bm25_weight=0.5。结果显示向量权重从1.0降到0.6的过程中忠实度和召回率持续提升,在0.6:0.4时达到峰值。等权(0.5:0.5)时召回率最高但精确度下降较多。综合考虑,0.6:0.4的权重分配在本数据集上取得了较好的平衡。

需要指出的是,这些参数的最优值与知识库规模、文档类型分布和查询模式相关。在当前18条文档分块的小规模知识库上的最优参数,在知识库扩充后可能需要重新调优。

### 5.5 重排序效果分析

Cross-Encoder重排序作为可选的精排阶段,其效果通过对比开启和关闭重排序时的指标变化来评估。

实验在混合检索配置下进行,开启重排序时retrieve_n设为20(粗排召回数量),top_n设为5(精排保留数量)。对比结果显示,开启重排序后上下文精确度有明显提升,从0.3667提升到约0.42,提升约14.5%。这是因为Cross-Encoder能够更准确地判断文档与问题的相关性,过滤掉粗排阶段误召回的文档。

重排序的代价是增加了约0.5-1秒的延迟(额外的API调用时间)。在延迟敏感的场景下可以选择关闭重排序,此时系统仍能通过RRF融合获得较好的基础效果。在准确性优先的场景(如金融专业问答)下建议开启重排序。

### 5.6 系统性能分析

本节从延迟和系统开销两个角度分析系统的运行性能。

#### 5.6.1 查询延迟分析

三种检索策略的延迟统计如下:

| 检索策略 | P50 (ms) | P95 (ms) | 平均 (ms) |
|---------|----------|----------|----------|
| vector | 7572 | 14201 | 7079 |
| bm25 | 6097 | 12031 | 6107 |
| hybrid | 6822 | 14647 | 7482 |

延迟数据包含完整的查询流程:检索→(可选重排序)→Prompt构造→LLM生成。其中LLM生成时间约5-6秒/条,占总延迟的主要部分。纯检索阶段的实际延迟在500ms以内。

纯BM25检索延迟最低(P50=6097ms),因为BM25基于倒排索引,检索速度快于向量检索的ANN搜索。混合检索延迟(P50=6822ms)介于两者之间,得益于双路检索的并行化——通过asyncio.gather(异步)或ThreadPoolExecutor(同步)同时执行两路检索,延迟取决于较慢的一路而非两路之和。

P95延迟明显高于P50,主要受两个因素影响:（1）网络波动导致API调用偶尔超时;（2）部分复杂查询需要更多的检索结果或更长的LLM生成时间。系统通过指数退避重试机制处理API超时,最大重试3次。

#### 5.6.2 异步化效果

系统实现了全链路异步化改造:Embedding调用、向量检索、BM25检索、重排序、LLM生成均支持async/await模式。混合检索中两路检索的并行化是最显著的优化点。

同步模式下,混合检索使用ThreadPoolExecutor将向量检索和BM25检索提交到两个线程并行执行,延迟从串行的两路之和降低到max(两路)。异步模式下使用asyncio.gather进一步减少了线程开销。实际测试中异步化将混合检索的纯检索阶段延迟降低约40%。

#### 5.6.3 缓存效果

语义级查询缓存在重复查询或语义相近查询场景下效果明显。在50条评估数据集上,因为每条查询只出现一次,缓存命中率较低。在实际使用场景中,当用户反复追问同一话题或使用不同表述方式提问时,缓存命中率会显著提升。缓存命中时响应延迟接近0(跳过检索和生成阶段),同时节省API调用额度。

## 第六章 总结与展望

### 6.1 本文工作总结

本文围绕金融领域知识问答的需求，设计并实现了一个基于混合检索的RAG知识库问答系统。系统从文档加载、文本分块、混合检索、重排序到答案生成的全链路进行了工程化实现，并通过实验验证了各模块的有效性。主要工作总结如下：

（1）设计并实现了BM25+向量混合检索与RRF融合方法。针对金融领域查询方式的多样性——用户既可能使用专业术语查询，也可能用自然语言描述需求——本文采用向量语义检索和BM25关键词检索双路并行检索的策略，通过RRF算法融合两路排名。实验结果表明，混合检索在忠实度上达到0.6224，相比纯向量检索提升9.9%，在上下文召回率上达到0.5455，提升23.1%。RRF算法只利用排名信息进行融合，天然兼容不同尺度的检索分数，实现简洁且效果稳定。

（2）实现了Cross-Encoder重排序、Query Rewriting和语义缓存等优化策略。重排序阶段通过Cross-Encoder对粗排结果进行二次筛选，上下文精确度提升约14.5%。Query Rewriting模块利用LLM将多轮对话中的指代性问题改写为独立查询，解决了多轮对话场景下检索失败的问题。语义级查询缓存通过Embedding余弦相似度匹配语义等价的查询，在命中时跳过检索和生成流程，降低了响应延迟和API调用成本。这些优化策略均通过配置开关独立控制，用户可根据实际场景灵活组合。

（3）构建了完整的金融RAG知识库问答系统。系统支持PDF研报、文本新闻、结构化问答等多种文档格式的自动加载和智能分块，提供段落优先和按标题分块两种策略。用户界面方面，Streamlit Web UI提供对话式交互和参数调节功能，FastAPI REST API提供程序化调用接口。系统集成RAGAS自动化评估框架，支持Faithfulness、Answer Relevancy、Context Precision、Context Recall四个核心指标的自动化评测。全链路异步化改造和双路检索并行执行保障了系统的响应性能。

### 6.2 未来展望

本文实现的系统仍有进一步优化和扩展的空间，未来可从以下几个方向改进：

（1）知识库规模扩充。当前实验使用的知识库仅包含18条文档分块，数据规模较小。在实际部署场景中，知识库需要涵盖数百篇研报和数千条新闻，这对检索效率和管理机制提出了更高要求。未来可引入增量索引构建和文档版本管理机制，支持大规模知识库的持续维护。

（2）Embedding模型优化。当前使用的通用Embedding-3模型在金融领域术语的编码上可能不是最优的。未来可尝试在金融语料上微调Embedding模型，或使用专门针对金融领域预训练的文本表示模型，提升向量检索的准确性。

（3）多模态文档支持。金融研报中包含大量表格、图表等非文本信息，当前系统仅能提取文本内容。未来可引入多模态文档解析能力，将表格数据结构化后纳入检索范围，提升对含表研报的处理能力。

（4）领域专用重排序模型。当前使用的通用Rerank API在金融领域数据上的表现有提升空间。未来可在金融问答数据集上微调Cross-Encoder模型，使其更好地理解金融术语和领域特定的语义关系。

（5）生产环境部署优化。当前系统面向开发和实验场景，在生产环境部署时需要增加用户认证、访问控制、API限流、日志审计等安全和运维功能。同时可引入Docker容器化部署方案，简化运维流程。

（6）评估体系的完善。当前评估数据集仅包含50条样本，类型覆盖有限。未来可扩充评估数据集规模，增加更多样化的金融问答场景，并引入人工评估作为自动化评估的补充，更全面地衡量系统在实际使用中的表现。
